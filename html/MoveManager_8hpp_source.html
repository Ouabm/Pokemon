92 // Pourquoi "extern const" est bien utilisé ici :
93 
94 // 1. Déclaration sans définition dans l'en-tête (header) :
95 // "extern const" permet de déclarer une constante sans la définir dans le fichier d'en-tête.
96 // Cela signifie que les objets Move (comme judgment, slash, etc.) sont définis ailleurs dans le code source.
97 // Cette approche permet de séparer les déclarations (interface) des définitions (implémentation),
98 // ce qui est une bonne pratique pour une meilleure organisation et un code plus lisible.
99 
100 // 2. Partage de constantes entre différents fichiers :
101 // Ces objets Move peuvent être utilisés dans plusieurs fichiers source différents. Grâce à "extern const",
102 // la même instance de chaque mouvement peut être partagée entre tous ces fichiers sans dupliquer leur définition.
103 // Cela permet d'éviter des redéfinitions et des conflits, et garantit que les mouvements restent cohérents dans tout le programme.
104 
105 // 3. Garantir l'immuabilité des objets :
106 // L'utilisation de "const" garantit que les valeurs des objets Move ne seront pas modifiées après leur initialisation.
107 // Cela améliore la sécurité du code en s'assurant que les mouvements définis dans le jeu (comme les dégâts, les types, etc.) ne changent pas par erreur.
108 // Cela optimise aussi le programme car les objets constants peuvent être plus facilement optimisés par le compilateur.
109 
110 // 4. Meilleure gestion de la mémoire :
111 // Étant donné que ces objets sont déclarés une seule fois (dans un fichier source),
112 // cela permet d'éviter la duplication des données et de réduire la consommation mémoire.
113 // Ainsi, un objet Move n'est pas copié dans chaque fichier qui l'inclut, mais il est utilisé via une référence constante.
